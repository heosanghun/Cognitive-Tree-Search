## 1. 문제 정의 — "KV-Cache Explosion"

### 발표 포인트

> **"LLM이 깊이 생각하게 만들고 싶은데, 생각할수록 메모리가 폭발합니다."**

- **System 1 vs System 2 사고**
  - System 1: 빠른 직관 (1회 forward pass) → 현재 LLM의 기본 모드
  - System 2: 깊은 추론 (탐색 + 되돌아가기 + 비교) → 인간의 수학 문제 풀이 방식

- **MCTS (Monte Carlo Tree Search)로 System 2를 구현하면?**
  - 트리의 각 노드마다 KV-Cache를 저장해야 함
  - depth 35에서 RTX 4090 (24GB) OOM 발생
  - 이것이 **KV-Cache Explosion** 문제

- **핵심 수치로 문제 크기 전달:**

| 상황 | KV-Cache 크기 |
|------|:------------:|
| Depth 10 | 220 MB |
| Depth 35 | 770 MB (+ 모델 16GB = OOM) |
| Depth 100 | 2.2 GB (불가능) |

> **한 줄 요약:** "트리가 깊어질수록 메모리가 선형으로 증가 → 깊은 추론이 원천적으로 불가능"

## 2. 핵심 아이디어 — "KV-Cache를 아예 없앤다"

### 발표 포인트

> **"KV-Cache를 줄이는 게 아니라, 근본적으로 필요 없게 만들자"**

- **기존 접근:** KV-Cache 압축, 양자화, 프루닝 → 본질적 한계 (여전히 O(depth))
- **CTS의 접근:** KV-Cache 자체를 제거 → **O(1) VRAM**

- **어떻게?**
  - 각 노드의 "상태"를 긴 토큰 시퀀스가 아니라 **고정 크기 잠재 벡터 z*** 로 표현
  - z*는 **Deep Equilibrium Model (DEQ)** 의 고정점 (fixed-point)
  - z*를 구하는 데 KV-Cache가 필요 없음 → **O(1)**

```
기존 MCTS:  노드 = [token₁, token₂, ..., tokenₙ] + KV-Cache  →  O(n)
CTS:        노드 = z* (고정 크기 벡터, K=64 tokens)          →  O(1)
```

> **비유:** "책 전체를 기억하는 대신, 핵심 요약 메모 한 장만 들고 다니는 것"

| 루프 | 역할 | 비유 |
|------|------|------|
| **Outer Loop (MCTS)** | "어디를 탐색할지" 결정 | 체스에서 수를 고르는 것 |
| **Inner Loop (DEQ)** | "한 수를 실제로 계산" | 각 수의 결과를 시뮬레이션 |

## 4. 핵심 기술 5가지

### 4-1. Deep Equilibrium Model (DEQ) — 논문 §4.2, Eq.(2)

**무엇인가:**
- 일반 신경망: 입력 → L개 레이어 순차 통과 → 출력
- DEQ: **같은 레이어를 무한히 반복**하여 **고정점(fixed-point)** z*에 수렴
  - z* = f(z*, 입력) — z*를 넣으면 z*가 나오는 점

**왜 중요한가:**
- L개 레이어의 출력을 **1개 레이어의 고정점**으로 대체
- 중간 활성값 저장 불필요 → **메모리 O(1)**
- Broyden 해법으로 10~15회 반복만에 수렴

**코드 위치:** `cts/deq/broyden_forward.py`, `cts/deq/transition.py`

---

### 4-2. L-Broyden Solver — 논문 §5.2

**무엇인가:**
- DEQ의 고정점 z*를 빠르게 찾는 준-뉴턴(quasi-Newton) 방법
- Newton 방법의 역 야코비안을 **rank-1 업데이트**로 근사
- FP32 버퍼로 수치 안정성 보장

**핵심 수치:**
- 평균 12회 반복으로 수렴
- 최종 잔차(residual): 3.42 × 10⁻³
- 수렴율: 97.3% (논문 부록 C)

**지도교수님 예상 질문:**
> "Broyden이 수렴하지 않으면?" → Anderson mixing 폴백 + 부모 J⁻¹ 상속으로 warm start

---

### 4-3. FAISS Latent Context Window — 논문 §4.4

**무엇인가:**
- 트리의 과거 노드 정보를 **잠재 벡터 z*** 로 저장
- 새 노드 확장 시, FAISS로 **의미적으로 가장 유사한 3개** 조상 검색
- 검색된 벡터를 soft prefix로 주입

**왜 중요한가:**
- 기존: 전체 대화 히스토리를 토큰으로 유지 → O(depth × seq_len)
- CTS: 벡터 인덱스로 검색 → **O(N) 킬로바이트**

**코드 위치:** `cts/latent/faiss_context.py`

---

### 4-4. Meta-Policy ν 벡터 — 논문 §2.3

**무엇인가:**
- 5차원 **신경조절 벡터** ν = [ν_val, ν_expl, ν_tol, ν_temp, ν_act]
- MCTS의 탐색 행동을 **동적으로 조절**

| 기호 | 조절 대상 | 직관적 의미 |
|------|----------|------------|
| ν_val | 상태 가치 평가 | "이 노드가 얼마나 유망한가" |
| ν_expl | 탐색 vs 활용 | "새로운 곳을 얼마나 탐험할까" |
| ν_tol | Broyden 정밀도 | "얼마나 정확하게 계산할까" |
| ν_temp | 모듈 라우팅 온도 | "전문가 모듈을 얼마나 집중할까" |
| ν_act | 조기 종료 임계값 | "언제 생각을 멈출까" |

**왜 중요한가:**
- 쉬운 문제 → ν_tol 높게 (대충 풀어도 됨), ν_act 낮게 (빨리 멈춤)
- 어려운 문제 → ν_tol 낮게 (정밀하게), ν_expl 높게 (넓게 탐색)
- PPO로 학습 → 문제별 최적 전략 자동 학습

**코드 위치:** `cts/policy/meta_policy.py`, `cts/types.py`

---

### 4-5. Gemma 4 E4B 42→19 모듈 파티션 — 논문 §5.1

**무엇인가:**
- Gemma 4 E4B의 42개 트랜스포머 블록을 **19개 기능 모듈**로 재분할
- Per-Layer Embeddings (PLE) 구조를 활용한 파라미터 공유
- Top-k 희소 라우팅으로 **3개 모듈만 활성화**

**왜 중요한가:**
- 42개 전부 실행하면 FLOPs 과다 → **3/19만 실행**하여 ~84% FLOPs 절약
- Iso-FLOP 제약 하에서 더 많은 트리 탐색 가능

**코드 위치:** `cts/routing/sparse_moe_ref.py`, `cts/backbone/gemma_adapter.py`

> **"같은 연산량 제약에서 CTS가 모든 벤치마크 SOTA"**

| 벤치마크 | CTS | Native Think | 개선폭 |

## 부록: 예상 질문 & 답변

### Q1. "DEQ가 수렴하지 않으면 어떻게 하나?"
> L-Broyden은 30회 반복 제한이 있고, 수렴 실패 시 Anderson mixing으로 폴백합니다.
> 실험에서 수렴율은 97.3%이며, 나머지 2.7%도 사용 가능한 근사 해를 반환합니다.
> 또한 부모 노드의 역 야코비안 J⁻¹을 상속하여 warm start하므로,
> 트리 탐색이 진행될수록 수렴 속도가 빨라집니다.

### Q2. "KV-Cache 없이 정보 손실은 없나?"
> K=64 잠재 토큰이 정보를 압축하며, 논문 §4.5에서 정보 보존율 91.4% 이상을 검증했습니다.
> FAISS 문맥 윈도우가 과거 히스토리를 의미적으로 검색하여 보완합니다.
> 실험 결과 (Table 2), KV-Cache 기반 방법보다 성능이 오히려 높습니다.

### Q3. "왜 Gemma 4 E4B인가? 다른 모델은?"
> Gemma 4 E4B는 Per-Layer Embeddings (PLE) 구조를 가진 SLM으로,
> 42개 블록이 파라미터를 공유하므로 DEQ의 "같은 함수 반복" 개념과 자연스럽게 맞습니다.
> 다른 모델 (LLaMA, Qwen 등)에는 PLE가 없어 추가 어댑터가 필요합니다.

### Q4. "RTX 4090으로 정말 충분한가?"
> 네, 논문의 핵심 주장입니다. CTS는 **단일 소비자용 GPU**에서 동작하도록 설계되었습니다.
> 모델 가중치: ~16 GB, CTS 오프헤드: ~0.7 GB, Vision/Audio 오프로드: -0.9 GB
> 총 ~16.7 GB로 24 GB VRAM 내에서 안정적으로 동작합니다.

### Q5. "Iso-FLOP이란? 왜 중요한가?"
> 동일한 총 연산량(MACs) 제약에서 비교하는 공정한 평가 프로토콜입니다.
> CTS는 트리 탐색으로 추가 연산을 사용하므로,
> 같은 연산 예산에서 기존 방법과 비교해야 공정합니다.
> CTS는 예산의 65%만 사용하고도 SOTA를 달성하여,
> "더 효율적으로 더 잘 추론한다"는 것을 증명합니다.

### Q6. "논문의 novelty는 무엇인가?"
> 1. **O(1) VRAM per node** — KV-Cache를 아예 제거한 최초의 MCTS 프레임워크
> 2. **DEQ + MCTS 결합** — 고정점 반복과 트리 탐색의 수학적 통합
> 3. **FAISS 잠재 공간 문맥 윈도우** — O(N) KB로 전체 트리 히스토리 접근
> 4. **Meta-Policy ν 벡터** — 5차원 신경조절 신호로 적응적 탐색
